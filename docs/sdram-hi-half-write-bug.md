# SDRAM upper-half-word write drop (task #146)

## Symptom

Every JTAG-Master 32-bit write to SDRAM commits only the lower 16 bits.
The upper 16 bits silently fail to commit; the SDRAM cell's upper
byte lane retains whatever was there before. Reproduces deterministically
across all addresses, all byte-enable patterns, and all clock speeds we
tested. See `nix run .#sdram-write-pattern-test` for an isolated
reproducer (25 / 29 fail on the current bitstream).

## What we ruled out

| Layer | Verdict | Evidence |
|---|---|---|
| Clash SDRAM adapter (`Riski5.Sdram`) | ✅ correct | `test/SdramSpec.hs` — 8 tests passing, including 32-bit SW + back-to-back SH writes |
| Clash JTAG-Avalon-Master (`Riski5.JtagAvalonMaster`) | ✅ correct | byte-stream → 32-bit Avalon assembly logic verified by inspection (`setByte`, `setBe`) |
| Hand-written CDC bridge (`riski5_sdram_cdc_bridge`) | ✅ correct | `sim/cdc_bridge_test.cpp` — 14 / 14 pass under Verilator with a behavioral SDRAM IP, including odd half-word indices and lo+hi pair writes |
| IP parameter mismatch | ✅ all match the chip | dataWidth=16, columnWidth=8, rowWidth=12, numBanks=4, bankWidth=2 → addressWidth=22, size=8MB. Matches IS42S16400-7TL datasheet. CL=3 conservative for 108 MHz (chip rated CL=3 at 143 MHz). |
| FPGA pin assignments | ✅ correct | `pkgs/riski5-core/Riski5.qsf` lines 304-323 match Terasic DE2 User Manual; DRAM_UDQM = PIN_Y5, DRAM_LDQM = PIN_AD2. |
| 3.3-V LVTTL IO standard | ✅ correct | applied to all DRAM_* including DQ[15:0] and UDQM |
| SDRAM clock phase shift | ✅ in place | -3 ns on DRAM_CLK output (commit a6df51b — task #132) |
| Clock-rate / timing slack | ✅ not the issue | `slowClock=true` build (30 MHz uniform) shows the same drop pattern; not a 108 MHz timing race |

## What's left

The remaining suspects are physical / IP-internal:

1. **Hardware fault on the chip's upper byte lane.** The DRAM_DQ[15:8] /
   DRAM_UDQM signal could be defective on the specific board (cracked
   trace, bad solder joint, dead chip pin). Easy to confirm with a
   scope or logic analyzer on DRAM_UDQM during a write — should pulse
   low while the controller is asserting DQ[15:8] for an upper-byte
   write.

2. **DDR I/O register timing** for DQ[15:8] specifically. The FPGA
   places DRAM_DQ in an I/O cell register; if the placement for the
   upper 8 lanes hits a timing constraint that the lower 8 don't, only
   the upper is corrupted. Quartus's `report_timing` against the
   DRAM_DQ paths could surface this.

3. **The Altera SDRAM Controller IP's auto-generated Verilog** has
   some parameter combination (or a default that doesn't match our
   chip) that disables upper-byte writes. Its source is generated by
   the Perl scripts in
   `share/altera13.0sp1/ip/altera/sopc_builder_ip/altera_avalon_new_sdram_controller/`
   and isn't shipped with the bitstream package. Could be reproduced
   by running `ip-generate` standalone and inspecting
   `riski5_sdram.v`.

## Recommended next step

Replace the Altera IP with a Clash-side SDR SDRAM controller (option
explicitly listed in CLAUDE.md / phase-1D as the fallback if the
Altera IP has issues we can't fix). The bus interface
(`Riski5.AvalonMm`) and the 32→16 width adapter (`Riski5.Sdram`) stay;
only the actual init/refresh/ACTIVATE/READ/WRITE/PRECHARGE FSM gets
re-implemented. Open-source references suitable for porting:

- `bitsavers.org` SDRAM controllers (Verilog or VHDL — port to Clash)
- Alex Forencich's verilog-axi sdram_controller
- Open iCE40 SDR controllers on GitHub

The replacement controller can be unit-tested directly (it's pure
Clash) against `sdramIpSim` — no JTAG-Master path needed for the
lower layers. Unit tests then rule out the IP-or-chip dichotomy
above.

## Test infrastructure available now

- `nix run .#sdram-write-pattern-test` — silicon-side reproducer; runs
  in seconds against any flashed bitstream that exposes the
  JTAG-Avalon-Master service.
- `sim/cdc_bridge_test.cpp` — Verilator + behavioral IP regression
  test for the CDC bridge. Run via `cd sim && nix shell ...` per
  `sim/README.md`.
- `test/SdramSpec.hs` — Haskell-side regression test for the
  SDRAM 32 ↔ 16 adapter. Already covers SW/LW round-trip and
  back-to-back SH writes.
