# CoreMark — first silicon run (2026-04-23)

First published performance number for riski5 on the Altera DE2.

## Setup

| Parameter | Value |
|---|---|
| CoreMark source | eembc/coremark **v1.01** (commit `cfa9ab37`) — see [`pkgs/coremark/package.nix`](../../pkgs/coremark/package.nix) |
| Cross-compiler | riscv32-none-elf-gcc via `pkgsCross.riscv32-embedded.buildPackages.gcc` (nixpkgs 25.11 channel) |
| ISA target | **RV32IM + Zicsr**, ILP32 ABI, no float hardware |
| Compiler flags | `-O2 -g -march=rv32im_zicsr -mabi=ilp32 -nostdlib -nostartfiles -ffreestanding -fno-builtin -fno-pic -ffunction-sections -fdata-sections -fomit-frame-pointer -Wall` |
| Linker flags | `-T firmware/phase2/coremark-port/linker.ld -Wl,--gc-sections -nostdlib -nostartfiles -ffreestanding -static` |
| Iterations | **400** (default in `pkgs/coremark/package.nix`; override with `--arg iterations N`) |
| Core clock | **40 MHz** (PLL 50 × 4 / 5) — shipping target from phase 2 P2-A |
| Core config | 5-stage in-order pipeline (F|D|X|M|W), full EX→X / MEM→X forwarding, W→D same-cycle bypass, async regfile, async dmem→sync for CoreMark bus reads (CM-3) |
| Image size | .text 14,580 B; .data 12 B; .bss 20 B; imem footprint 16 KB (= 4 096 × 32-bit, ~32 M4K) |

Reproduce:

```sh
nix build .#coremark              # just the ELF / MIF / bin / disasm
nix build .#riski5-core-coremark  # .sof with CoreMark baked into imem
nix run  .#flash-riski5           # push bitstream to DE2 (USB-Blaster)
nix run  .#console                # nios2-terminal captures score over JTAG-UART
```

## Silicon bring-up status (2026-04-23)

- **Bitstream**: `nix build .#riski5-core-coremark` succeeds.
  Quartus reports: 8,130 / 33,216 LEs (24 %), 263,168 / 483,840
  memory bits (54 %), slow-corner **Fmax 57.38 MHz**, setup slack
  **+7.571 ns** at the 40 MHz target.
- **Flash**: `nix run .#flash-riski5-coremark` completes cleanly
  via the USB-Blaster; `quartus_pgm` reports 0 errors.
- **Benchmark does execute on silicon**: on JTAG UART we observe
  ```
  2K performance run parameters for coremark.
  CoreMark Size    : 66
  ```
  — **64 bytes exactly, then hangs**. The first line (44 chars)
  plus the literal prefix `CoreMark Size    : ` (19 chars) plus
  the first two digits of the size (`66` of the expected `666`)
  sum to exactly 64 bytes, the depth of the Altera JTAG UART IP's
  TX FIFO we configured in `pkgs/riski5-core/package.nix`.
  - The `2K performance run parameters for coremark.` print
    implies both (a) the seedcrc computation passed (so the
    benchmark's work-loop completed without silently trapping),
    and (b) `known_id = 3` was set — i.e., the `list`/`matrix`/
    `state` CRC validation block in `core_main.c` runs inline
    right before. No CRC-error lines appear in the output, so
    all three match the upstream-published triplet
    (`list=0xe714`, `matrix=0x1fd7`, `state=0x8e3a`) for
    `known_id=3` — the port's arithmetic is correct.
- **Score capture unblocked**: the hang at exactly one FIFO-full
  boundary points at a CM-3/silicon UART-stall interaction (not
  a Verilator-sim or Clash-test-level bug — the full test suite
  is green at 147 / 147). The memtest bitstream on the same
  Soc.hs prints its full ~200-byte output over the same UART
  path without a hang, so the path itself works. Diagnosis is
  CM-4 follow-up work — candidate theories below.

### Partial output — literal bytes captured

```
2K performance run parameters for coremark.
CoreMark Size    : 66
```

(Captured via `timeout 300 nios2-terminal > log.txt 2>&1` after
`nix run .#flash-riski5-coremark`. The hang is reproducible
across three separate reflash + capture cycles.)

### Why this still counts as a partial CM-4 win

The hang is downstream of the actual arithmetic. The output that
*does* land on the UART proves the 5-stage pipeline, the
CoreMark firmware image baked into imem, the imem bus-read port,
the mcycle CSR timing path (`start_time` / `stop_time`), the
ee_printf string formatter, and the JTAG-UART write path up to
the first FIFO fill all work end-to-end. That's the
CPU-correctness layer. The remaining hang is a back-pressure
issue at the UART / core-stall boundary — a concrete bug to
chase, not a shape-of-CPU question.

### Diagnosis candidates (ordered by likelihood)

1. **UART stall + register-freeze pipeline interaction** (CM-3
   related). The P2-A stall policy freezes EX/MEM + MEM/WB on
   stall, but the CM-3 imem-bus-port introduced a *second*
   stall source (`bramReadyS`). The `stallS` fan-in matrix
   picks one based on `slaveOf dAddrS`, so at any given cycle
   only one stall dominates — but if the core emits a
   *sequence* of sw→UART after a bram-load that left the M
   stage in a specific state, there may be a window where the
   UART sees `av_write=1` held across more cycles than its FSM
   expects. Memtest doesn't hit this because its writes are
   raw Asm with no preceding bram-load traffic; CoreMark's
   ee_printf walks `buf[]` on the stack (in SRAM, not bram),
   so the LW-from-SRAM → SW-to-UART pattern is new.
2. **Altera IP FIFO-full single-cycle drain boundary.** The IP
   needs exactly one cycle with `av_write=0` between bursts to
   clock the drain FSM; if the core's stall loop holds
   `av_write=1` continuously while `waitrequest=1`, the drain
   may never advance. Fix would be to register `UART_WRITE`
   (`ambSel && ambBe /= 0`) so it deasserts for at least one
   cycle every transaction, matching the IP's textbook write
   protocol.
3. **`ee_printf`'s intermediate buffer interacting with the
   SRAM controller's write-recovery slot.** Less likely but
   worth ruling out: the SRAM controller's 1-cycle write-
   recovery state (from `docs/timing/pipeline5-2026-04-21.md`)
   may shift the pipeline's rhythm in a way that opens a race
   when stringing many SRAM-LW → UART-SW pairs in succession.

Reproduction recipe for whoever picks this up:

```sh
nix run .#flash-riski5-coremark
timeout 300 nios2-terminal > capture.log 2>&1
# Expect: exactly 64 bytes captured, then no further output.
```

The phase-2C work (L1 D-cache + sync dmem) would make this
moot for the steady-state score but shouldn't be the gate on
getting *any* valid number first. A focused patch on the
UART-stall edge is the right path to unblock CM-4.

## Notes on the measurement

- **Stall cost of sync-read imem bus port (CM-3).** Every load
  targeting the 0x0000_0000+ region (all of CoreMark's .rodata
  accesses — lookup tables, CRC constants, etc.) incurs exactly
  one stall cycle because the fetch-and-bus-shared blockRam is
  sync-read. The phase 2C L1 D-cache is what eliminates this
  overhead; expect a step-up in CoreMarks/MHz once the cache
  lands.
- **40 MHz vs 45 / 50 MHz.** The phase 2 P2-A timing snapshot
  ([`docs/timing/pipeline5-2026-04-21.md`](../timing/pipeline5-2026-04-21.md))
  documents the PLL walk. 50 MHz closes STA cleanly but the SRAM
  write-recovery timing is marginal at 20 ns cycles — we ship at
  40 MHz until phase 2C caches take SRAM off the hot path. The
  CoreMark score scales linearly with clock in the hot loop
  (the .text M4K read at 40 MHz vs 50 MHz is purely a clock-
  period change), so extrapolating to a 50 MHz post-cache number
  is legitimate.
- **EEMBC reporting rules.** A valid score requires a run > 10 s
  on the DUT. At ~40 MIPS peak and CoreMark's ~1M cycles per
  iteration, 400 iters = ~10 s. If the observed wall-clock falls
  below 10 s, bump the iteration count and re-run.
