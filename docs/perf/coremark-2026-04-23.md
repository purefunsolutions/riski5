# CoreMark — first silicon run (2026-04-23)

First published performance number for riski5 on the Altera DE2.

## Setup

| Parameter | Value |
|---|---|
| CoreMark source | eembc/coremark **v1.01** (commit `cfa9ab37`) — see [`pkgs/coremark/package.nix`](../../pkgs/coremark/package.nix) |
| Cross-compiler | riscv32-none-elf-gcc via `pkgsCross.riscv32-embedded.buildPackages.gcc` (nixpkgs 25.11 channel) |
| ISA target | **RV32IM + Zicsr**, ILP32 ABI, no float hardware |
| Compiler flags | `-O2 -g -march=rv32im_zicsr -mabi=ilp32 -nostdlib -nostartfiles -ffreestanding -fno-builtin -fno-pic -ffunction-sections -fdata-sections -fomit-frame-pointer -Wall` |
| Linker flags | `-T firmware/phase2/coremark-port/linker.ld -Wl,--gc-sections -nostdlib -nostartfiles -ffreestanding -static` |
| Iterations | **600** (default in `pkgs/coremark/package.nix`; override with `--arg iterations N`). Bumped from 400 after the first silicon run measured 8.98 s, under EEMBC's 10 s validity threshold. |
| Core clock | **40 MHz** (PLL 50 × 4 / 5) — shipping target from phase 2 P2-A |
| Core config | 5-stage in-order pipeline (F|D|X|M|W), full EX→X / MEM→X forwarding, W→D same-cycle bypass, async regfile, async dmem→sync for CoreMark bus reads (CM-3) |
| Image size | .text 14,580 B; .data 12 B; .bss 20 B; imem footprint 16 KB (= 4 096 × 32-bit, ~32 M4K) |

Reproduce:

```sh
nix build .#coremark              # just the ELF / MIF / bin / disasm
nix build .#riski5-core-coremark  # .sof with CoreMark baked into imem
nix run  .#flash-riski5           # push bitstream to DE2 (USB-Blaster)
nix run  .#console                # nios2-terminal captures score over JTAG-UART
```

## Silicon bring-up status (2026-04-23 — first valid score)

- **Bitstream**: `nix build .#riski5-core-coremark` succeeds.
  Quartus reports: **8,142 / 33,216 LEs (25 %)**, 263,168 /
  483,840 memory bits (54 %), slow-corner **Fmax 54.17 MHz**,
  setup slack **+6.540 ns** at the 40 MHz target.
  (Fmax dropped slightly from 57.38 MHz pre-CM-4 because
  implementing `mcycle` as a free-running 32-bit counter adds
  a 32-bit adder and 32 FFs to the CSR cone.)
- **Flash**: `nix run .#flash-riski5-coremark` completes cleanly
  via the USB-Blaster; `quartus_pgm` reports 0 errors.
- **Silicon score (first EEMBC-valid run)**: below.

The `CoreMark 1.0 : <score> / ...` summary line (EEMBC's canonical
one-liner) is only emitted when `HAS_FLOAT == 1`; our port sets it
to 0 (no FPU), so we compute the score from `Total ticks` /
`Iterations`. The `Iterations/Sec : 46` field is the integer-divided
approximation (`600 / 13 = 46`); the precise value comes from the
tick counter.

| Wall-clock time (ticks / 40 MHz) | **13.463 s** |
| Iterations | 600 |

| **CoreMarks / MHz** | **1.114** |
| Correct-operation check | ✓ (crclist + crcmatrix + crcstate all match published `known_id = 3` triplet) |
| Validity (≥ 10 s run) | ✓ 13.46 s |

**Plain reading**: riski5's 1.114 CoreMarks/MHz sits just above
PicoRV32 small (0.516) and VexRiscv Min (0.79), and at ~55 % of
VexRiscv Full (2.02). The 5-stage pipeline puts us ahead of the
unpipelined PicoRV32 baseline; absence of an I-cache / D-cache
and the one-cycle SlaveBram stall per .rodata load hold us
below Rocket / VexRiscv Full — both milestones scheduled for
phase 2C in [`docs/core-family.md`](../core-family.md) §4.3.

## Known errata from this run

- **`0xFF` pad byte on some `CoreMark Size  : ...` lines.** The
  `ee_printf` pipeline stages a 256-byte local buffer on the
  stack (SRAM). The first-line capture from an earlier CM-4
  partial run showed `CoreMark Size    : 66` (truncated to the
  first 64 bytes of the run — exactly one JTAG UART TX-FIFO —
  because raw back-to-back `sw` to `0x1000_0000` with no
  read-back cycle between them lets the Altera IP's drain FSM
  get stuck). Fix landed: `core_portme.c` now polls the
  CONTROL register's WSPACE field (bits [31:16]) before each
  write, inserting a `lw` transaction between writes. The
  `lw` naturally toggles `av_write`/`av_read` at the IP
  boundary, giving it the cycle it expects. After this fix
  the output stream runs to completion reliably.
- **`mcycle` was previously not implemented.** `Riski5.CSR`'s
  `readCsr` fall-through returned 0 for every unrecognised
  CSR, and `mcycle` (`0xB00`) was in that set. CM-4 added
  `cMcycle :: BitVector 32` to the `Csrs` record, a free-
  running increment in `Core.hs` (every core clock, not
  per-retire), and read/write paths for `0xB00`. 32-bit is
  enough for any CoreMark run below ~107 s at 40 MHz; `mcycleh`
  remains zero (CM-5 follow-up if we ever need > 107 s).

## Notes on the measurement

- **Stall cost of sync-read imem bus port (CM-3).** Every load
  targeting the 0x0000_0000+ region (all of CoreMark's .rodata
  accesses — lookup tables, CRC constants, etc.) incurs exactly
  one stall cycle because the fetch-and-bus-shared blockRam is
  sync-read. The phase 2C L1 D-cache is what eliminates this
  overhead; expect a step-up in CoreMarks/MHz once the cache
  lands.
- **40 MHz vs 45 / 50 MHz.** The phase 2 P2-A timing snapshot
  ([`docs/timing/pipeline5-2026-04-21.md`](../timing/pipeline5-2026-04-21.md))
  documents the PLL walk. 50 MHz closes STA cleanly but the SRAM
  write-recovery timing is marginal at 20 ns cycles — we ship at
  40 MHz until phase 2C caches take SRAM off the hot path. The
  CoreMark score scales linearly with clock in the hot loop
  (the .text M4K read at 40 MHz vs 50 MHz is purely a clock-
  period change), so extrapolating to a 50 MHz post-cache number
  is legitimate.
- **EEMBC reporting rules.** A valid score requires a run > 10 s
  on the DUT. At ~45 CoreMark 1.0 and CoreMark's ~1M cycles per
  iteration, 600 iters = 13.46 s on silicon — comfortably above
  the floor. The 400-iter first attempt came in at 8.98 s and
  tripped `ERROR! Must execute for at least 10 secs for a valid
  result!`; bumping the default in `pkgs/coremark/package.nix`
  to 600 fixed it.
