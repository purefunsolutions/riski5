## Critical path — slow model

Every top-100 setup path has the same shape:

```
From:     altsyncram imem address register (ram_block1a0~porta_address_regN)
To:       riski5:u_riski5|regs[908]            (regfile write-port bit)
Delay:    30.287 – 30.293 ns  (data delay)
Clock skew:        −0.070 ns
Slack @ 33.33 ns:  +3.012 ns
```

That's the **pipelineless single-cycle datapath** on one clock
period:

1. Imem address register (clocked by `clk30`, drives M4K BRAM).
2. Imem M4K lookup → 32-bit instruction out.
3. `Riski5.Decode` → `Instr`.
4. Regfile read (combinational, async-read register array).
5. ALU / MulDiv / CSR FUs (combinational).
6. Memory bus decoder (combinational; for ALU ops the memory arm is
   inert).
7. Writeback mux (combinational).
8. Regfile write-port (clocked by `clk30`, latches on next edge).

Every stage after the imem M4K is LUT-mapped combinational logic.
The path ends in `regs[908]` (regfile bit flat-index 908 = reg 28,
bit 12), suggesting the worst path converges on high bits of a
specific register the design exercises — but the overall shape
is uniform across the top-100.

## What this implies for phase 1E

The plan scope-guards phase 1E:

> Phase 1E is **exploratory** — if we plateau below a target we can
> live with, we stop and move on. The goal is to learn what the
> design costs at speed, not to force a specific number.

At slow corner, the design is physically limited to **~33 MHz**
because the entire instruction → regfile path settles in 30 ns and
no amount of Quartus tuning changes that materially (already at
`OPTIMIZATION_TECHNIQUE SPEED`, physical-synthesis retiming +
register duplication + extra effort, standard fit).

Shortening the cone further requires **pipelining** — e.g.:

- Register after decode (split fetch | decode from execute).
- Register after ALU (split execute from memory).
- Register after regfile read (split decode from execute).

Any of those cuts the combinational chain but changes the
single-cycle architecture into a 2–3 stage pipeline. That's the
phase-2 P2-A work explicitly called out in
[`docs/core-family.md`](../core-family.md).

## Local experiments already ruled out

| Idea | Why rejected |
|---|---|
| Move regfile to M4K | Would insert an extra cycle of read latency into the critical path cycle (currently async LE-mapped for the pipelineless design). Returns when we pipeline. |
| Register the imem output | Effectively *is* pipelining — splits F from X. |
| Widen the 5-stage barrel shifter | Shifter already at 5 log-stages, 4-LUT-friendly; not the bottleneck. |
| Increase Quartus effort | Already at EXTRA physical-synthesis + SPEED + retiming. |
| Reduce MulDiv iteration count | MulDiv is iterative and not on the critical path per the STA report. |

## Board reality — what actually works

At the current 30 MHz target clock:
- **Slow corner (85 °C, slowest silicon)**: +3.01 ns slack — design
  meets timing with real margin, signed-off by Quartus STA.
- **Fast corner (0 °C, best silicon)**: +19.16 ns slack — could run
  at ~70 MHz, but that's not guaranteed across corners.

The 30 MHz sign-off leaves the PLL configured at `50 × 3 / 5`.
Pushing to 40 MHz would require `50 × 4 / 5 = 40 MHz` → 25 ns
period, which the 30.3 ns critical path misses by 5.3 ns at slow
corner. The board might run it anyway at room-temp with nominal
silicon, but it wouldn't be a safe sign-off — and the hobby
context doesn't need the extra 10 MHz urgently.

## Decision — T41–T44

Phase 1E closes here. The plan's T41–T44 steps assumed Quartus
effort changes or a higher PLL target would move the needle; the
STA data shows they can't.

- **T40 done** (this document).
- **T41 skipped** — no PLL retarget that closes timing is available.
- **T42 done** (critical path identified, documented above,
  declared pipelining-only).
- **T43 / T44 skipped** — nothing changes at the current single-
  cycle architecture, so there's no higher-clock run to validate.

The next meaningful performance work is **phase 2 P2-A pipelining**
— see `docs/core-family.md`'s Tiny-tier roadmap. After a 2-stage F
| XM split, the imem-address → regfile cone breaks into two shorter
cones; STA will reveal the next bottleneck, which is likely the
ALU adder + branch-compare plus writeback mux.
