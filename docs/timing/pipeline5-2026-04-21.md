# Timing snapshot — 2026-04-21 (end of phase 2 P2-A)

Post-pipelining state after the phase 2 P2-A rewrite moved the
core from pipelineless-style 2-stage F+X to the classic 5-stage
F|D|X|M|W with full forwarding. Taken straight after the pipeline
commit landed green (commit `10fa187`), built via the same Nix
flow as the phase-1D baseline.

Bitstream: the current `nix build .#riski5-core` output from
commit `10fa187` (P2A-2).

The target core clock moves from 30 MHz straight up to 50 MHz —
we iterated the PLL (30 → 40 → 45 → 50 MHz), flashed and silicon-
verified each step, and stopped at 50 MHz because we're 91 % of
slow-Fmax there and any further push needs a shorter X cone
(phase 2B / 2C work). The +613 LE uptick vs baseline is
Quartus trading area for speed at the tight target; at 40 MHz
the same design fit in 10,955 LEs.

## What changed

Phase 1D's 30-ns critical cone
(`imem_addr_reg → decode → rf read → ALU → mem → wb mux →
regfile_write`) was the whole pipelineless datapath in a single
clock. Splitting it across five pipe registers — IF/ID, ID/EX,
EX/MEM, MEM/WB — and routing EX→X and MEM→X forwarding paths
bought us:

- __~15 ns off the worst path.__ The critical cone dropped from
  30.287 ns (baseline) to ~18.6 ns (after PLL bump — Quartus
  re-optimises for the tighter target).
- __+20 MHz of Fmax headroom.__ Slow-model Fmax 53.62 MHz vs
  baseline 32.98 MHz.
- __+10 MHz of core clock.__ PLL retargeted from 30 MHz to
  40 MHz (`50 × 4 / 5` instead of `50 × 3 / 5`); slack at that
  target is +6.35 ns, comfortable margin on slow-85 °C corner.
- __−432 LEs.__ The X stage's combinational cone shrank because
  pipelining let Quartus LUT-share between D and X stages, and
  the no-longer-critical-cone paths got optimised for area.

## Critical path — slow model

```
From:     riski5:u_riski5|idExS[*]  (X-stage input)
To:       riski5:u_riski5|c$$j1_case_alt_1[0]~OTERM*  (handleInstr dispatch)
Delay:    ~18.6 ns  (data delay)
Clock skew:        ~−0.01 ns
Slack @ 25.00 ns (40 MHz): +6.350 ns
```

Before the PLL bump (STA run with 30 MHz target), the worst path
from the same source cone was `idExS[134] → 20.438 ns →
c$$j1_case_alt_1[0]~11_OTERM79` with +12.923 ns slack — so the
bump cost only 3.3 ns of slack and gained 10 MHz of core clock.

The source-side `idExS[134]` and sibling bits (`idExS[135]`,
`exMemS[585]`, `exMemS[586]` — all in the same 12.9–13.0 ns
slack band) point at the X stage's combinational cone feeding
the EX/MEM pipe-register inputs. Shape: `ID/EX flip-flops →
forwarding mux → ALU + branch compare + CSR logic →
load-extension mux + writeback mux → EX/MEM flip-flops`.

The endpoint (`c$$j1_case_alt_1[0]~11_OTERM79`) is
Clash-generated glue for the `handleInstr` case split — i.e. the
dispatch over the `Instr` GADT constructors. That's roughly what
we'd predict: every in-flight instruction carries control signals
through the same dispatch, so its fan-in is wide (every
instruction kind's writeback path joins here).

Compared to the baseline, the X cone is now purely __execute-stage
combinational__ — no decode or regfile read in the cone. That
matches the design intent of 5-stage pipelining.

## What this buys for real-world use

At 30 MHz, a pipelineless single-cycle core and a pipelined 5-stage
core both retire at 1 instruction / clock in steady state, so IPC
is the same. The difference is __how much clock we can push__:

- __Pipelineless.__ Capped at ~33 MHz on Cyclone II by the whole
  datapath's worst path.
- __Pipelined.__ Capped at ~49 MHz by the X stage's worst path.
  The D / M / W stages are all faster, so splitting X further (or
  moving the memory access out of X into a sync-D$ M stage) would
  raise the ceiling again.

Taken-branch penalty rose from 1 cycle (pipelineless with
squashNext) to 2 cycles (5-stage with flush + flushPrev). That's
the cost-of-pipelining every RISC handbook warns about; on
branch-light code like the hello-firmware test loops it's
invisible. A branch predictor (even static BTFN) would recover
those cycles on loop backedges — phase 2C follow-up.

## Decision — PLL retargeting

Walked through four PLL targets; each step flashed + silicon-
verified on the DE2 with the phase-1 Hello firmware. Outcome
table:

| Target | PLL ratio (50 MHz × M / N) | Period | Slow-85 °C slack | Slow-Fmax reported | Silicon |
|---:|---|---:|---:|---:|---|
| 30 MHz | 3 / 5 | 33.33 ns | +12.923 ns | 49.00 MHz | ✓ green |
| 40 MHz | 4 / 5 | 25.00 ns | +6.350 ns | 53.62 MHz | ✓ green |
| 45 MHz | 9 / 10 | 22.22 ns | +3.956 ns | 54.75 MHz | ✓ green |
| **50 MHz** | **10 / 10** | **20.00 ns** | **+1.781 ns** | **54.89 MHz** | **✓ green (shipping)** |
| 55 MHz | 11 / 10 | 18.18 ns | —  | — | not attempted — above slow-Fmax ceiling |

We stop at 50 MHz. Fmax keeps creeping up each attempt because
Quartus reroutes for tighter targets (spending more LEs on
faster paths — 36 % utilisation at 50 MHz vs 33 % at 40 MHz),
but it's diminishing returns: at 50 MHz we're 91 % of the
reported slow-Fmax, slack is under 2 ns, and the next jump to
55 MHz needs the slow-85 °C Fmax to reach ≥ 55 MHz — which
this design's X stage can't do without being shortened.

Any further core-clock gain is a phase 2B / 2C concern:

- **2B.** `regfileAsync → regfileSync` (M4K regfile, 1-cycle
  read latency) takes the rs1 / rs2 read out of the D-stage
  combinational path and lets Quartus re-optimise X downstream
  of it. +2 M4K, roughly −300 LEs, probably −2..−3 ns on the X
  cone.
- **2C.** Sync dmem + 1 KB D$ moves the `extendLoad` byte-lane
  mux out of X into its own M stage, shortening the X cone
  further. Combined with 2B, slow-Fmax might reach 65-70 MHz.

The Tiny-tier roadmap in
[`docs/core-family.md`](../core-family.md) §4.3 already carries
both of those as scheduled work; this commit just captures the
PLL decision + ceiling number for the current X-in-one-stage
shape.

## Ship-now vs follow-up

__Ship now.__ 5-stage pipeline at 50 MHz clock, ~54.9 MHz
achievable, +66.4 % Fmax over the phase-1 baseline, 147 / 147
Clash tests green, phase-1 Hello firmware prints OK end-to-end
on DE2 silicon at the new clock.

__Follow-up for phase 2B / 2C.__

- Swap `Riski5.Regfile.regfile` (async LE) → `regfileSync` (M4K).
  Saves ~300 LEs, consumes 2 M4K. Read latency moves to D-stage
  issue → X-stage capture — needs the ID/EX reg to carry
  addresses instead of data plus a regfile-output forwarding
  mux in X. Separate work; not needed to ship P2A.
- Move dmem to sync BRAM (blocks naturally integrate with caches
  in phase 2C). Splits the current X-stage combinational cone
  by moving the load-extension logic to M, unlocking further
  Fmax gains.
- Push PLL to 45 MHz once one of the two items above has shortened
  the X cone enough that the +1.7 ns slack estimate is a real
  margin rather than a tight fit.

The Tiny-tier roadmap in
[`docs/core-family.md`](../core-family.md) §4.3 folds all three
of those into phase 2B (M-ext polish + minor forward work) and
phase 2C (I\$/D\$ + Zba/Zbb).
