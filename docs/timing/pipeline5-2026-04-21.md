# Timing snapshot — 2026-04-21 (end of phase 2 P2-A)

Post-pipelining state after the phase 2 P2-A rewrite moved the core
from pipelineless-style 2-stage F+X to the classic 5-stage F|D|X|M|W
with full forwarding. Built via the same Nix flow as the phase-1D
baseline.

__Shipping target__: **40 MHz** (PLL `50 × 4 / 5 = 40 MHz`). See the
"SRAM-limited PLL ceiling" section below for why we pulled back from
the 50 MHz that STA said closes fine.

## Summary — vs the phase-1D baseline

| Metric | Baseline (phase 1D, 2-stage) | Phase 2 P2-A, 5-stage @ 40 MHz (shipping) | Δ |
|---|---:|---:|---:|
| Total logic elements | 11,387 / 33,216 (34 %) | **10,955 / 33,216 (33 %)** | −432 LEs |
| Total memory bits | 31,744 | 62,976 | +31 232 (see §M4K note) |
| **Slow-model Fmax** | 32.98 MHz | **53.62 MHz** | **+62.6 %** |
| Worst-case setup slack (slow-85 °C) | +3.012 ns | **+6.350 ns** | +3.3 ns |
| Target core clock | 30 MHz (PLL 50 × 3 / 5) | **40 MHz (PLL 50 × 4 / 5)** | +33.3 % |

__M4K note__: the 8 KB imem (2048 words for the phase-2 P2-A BIOS

a timing change, just a firmware size change orthogonal to the core
rewrite.

## What changed

Phase 1D's 30-ns critical cone
(`imem_addr_reg → decode → rf read → ALU → mem → wb mux →
regfile_write`) was the whole pipelineless datapath in a single
clock. Splitting it across five pipe registers — IF/ID, ID/EX,
EX/MEM, MEM/WB — and routing EX→X and MEM→X forwarding paths
bought us:

- __~15 ns off the worst path.__ The critical cone dropped from
  30.287 ns (baseline) to ~18.6 ns.
- __+20 MHz of Fmax headroom.__ Slow-model Fmax 53.62 MHz vs
  baseline 32.98 MHz.
- __+10 MHz of core clock shipping.__ PLL retargeted from 30 MHz to
  40 MHz; slack at that target is +6.35 ns.
- __−432 LEs.__ The X stage's combinational cone shrank because
  pipelining let Quartus LUT-share between D and X stages.

## Critical path — slow model

```
From:     riski5:u_riski5|idExS[*]  (X-stage input)
To:       riski5:u_riski5|c$$j1_case_alt_1[0]~OTERM*  (handleInstr dispatch)
Delay:    ~18.6 ns  (data delay)
Slack @ 25.00 ns (40 MHz): +6.350 ns
```

The worst paths all come from the ID/EX pipeline-register outputs
into the X stage's combinational cone (ID/EX FF → forwarding mux →
ALU + branch compare + CSR logic → load-extension mux → writeback
mux → EX/MEM FF). The endpoint (`c$$j1_case_alt_1[0]~OTERM*`) is
Clash-generated glue for the `handleInstr` case split — i.e. the
dispatch over the `Instr` GADT constructors; its fan-in is naturally
wide because every instruction kind's writeback path joins there.

The X cone is now purely __execute-stage combinational__ — no decode
or regfile read in it, unlike phase 1D.

## SRAM-limited PLL ceiling — why 50 MHz STA closes but silicon
## doesn't ship there

Walked the PLL through four targets, flashing + silicon-verifying
the phase-1 Hello firmware at each:

| Target | PLL ratio | Period    | Slack    | Fmax     | Hello firmware |
|---:|---|---:|---:|---:|---|
| 30 MHz | × 3 / 5   | 33.33 ns  | +12.92 ns | 49.00 MHz | ✓ green     |
| 40 MHz | × 4 / 5   | 25.00 ns  | +6.35 ns  | 53.62 MHz | ✓ green     |
| 45 MHz | × 9 / 10  | 22.22 ns  | +3.96 ns  | 54.75 MHz | ✓ green     |
| 50 MHz | × 10 / 10 | 20.00 ns  | +1.78 ns  | 54.89 MHz | ✓ green     |

Hello firmware happily ran at every target up to 50 MHz. But when
the phase-2 P2-A BIOS memory test (see `firmware/phase1/MemTest.hs`)
landed — which walks __every__ SRAM address with back-to-back SW/LW
pairs — the 50 MHz build started returning bogus readback values
after a handful of words, while the same bitstream rebuilt at 40 MHz
completed the entire SRAM + SDRAM walk cleanly.

This is a __silicon-level write-recovery timing issue__ at the
SRAM pins, not a core pipeline bug:

- The off-chip SRAM chip (IS61LV25616-class, DE2 board) has
  something like 10–12 ns write-pulse-width requirement. Our
  controller's write-pulse and write-recovery states are each
  one core-clock cycle long.
- At 50 MHz (20 ns per cycle), the effective pulse at the chip
  pins — after FPGA I/O register Tco + pad + trace delay — is
  marginal (< 12 ns).
- At 40 MHz (25 ns per cycle) the same pulse is ~17 ns, comfortably
  inside spec. At 30 MHz it's ~27 ns, well inside.

The phase-1 Hello firmware's one-off SRAM test at a single address
happened to fit inside the margin; stressing the pins with thousands
of consecutive writes at 50 MHz exposed it.

Raising the PLL past 40 MHz without fixing SRAM write timing would
require one of:

1. Extending the SRAM controller's write-pulse / recovery states
   to 2 core-clock cycles each at 50 MHz (doubles SW latency —
   OK if we're fast enough overall).
2. Registering the SRAM pins through dedicated I/O FFs in a
   phase-shifted clock domain (standard trick for SDR-style
   off-chip memory).
3. Replacing the IS61LV25616 SRAM controller with one that
   auto-adjusts pulse width to the target clock (or uses the
   SDRAM-style Avalon-MM async memory wrapper from Altera).

None of those are blocking for P2-A's "pipelined 5-stage core on
DE2 silicon" milestone, so we ship at 40 MHz and pin 50 MHz as a
phase-2C follow-up once we have caches fronting the SRAM (cached
SRAM access doesn't hit the pins on every CPU load/store).

## Phase 2 P2-A BIOS memory test result

Built with the new 5-stage core + 40 MHz PLL:

```
MEMTEST BEGIN
SRAM : begin
SRAM : addr OK
SRAM : ~addr OK
SRAM : AAAA OK
SRAM : 5555 OK
SRAM : ALL OK
SDRAM: begin
SDRAM: addr OK
SDRAM: ~addr OK
SDRAM: AAAA OK
SDRAM: 5555 OK
SDRAM: ALL OK
MEMTEST ALL OK
```

(UART output may be interleaved by the 64-byte TX FIFO + host-side
buffering, but every address in both regions gets written and
verified.)

- __SRAM__: 128 K words × 4 patterns × 2 passes = 1 M accesses in
  a few hundred ms.
- __SDRAM__: 2 M words × 4 patterns × 2 passes = 16 M accesses in
  a few seconds.

Patterns: address-as-data, ~address (inverted-address), 0xAAAAAAAA
(even bits), 0x55555555 (odd bits). Each pattern runs a full write
pass followed by a full verify pass; the verify pass catches
cell-to-cell cross-talk, address-decoder aliasing, and SDRAM refresh
errors that the inline write-then-read can't see.

## Ship-now vs follow-up

__Ship now.__ 5-stage pipeline at 40 MHz clock, full memtest green
on both SRAM + SDRAM, 147 / 147 cabal tests green, Hello firmware +
phase-1 smoke tests all green on the same bitstream. Every existing
silicon test passes without rework.

__Follow-up for phase 2B / 2C.__

- Swap `Riski5.Regfile.regfile` (async LE) → `regfileSync` (M4K).
  Saves ~300 LEs, consumes 2 M4K. Read latency moves to D-stage
  issue → X-stage capture.
- Move dmem to sync BRAM + add the 1 KB L1 D$. Splits the X-stage
  combinational cone (load-extension into M) and takes SRAM /
  SDRAM off the CPU-load-per-instruction hot path — which is
  exactly what lets the PLL go past 40 MHz without tripping the
  SRAM write-recovery timing.
- 45 MHz PLL target after the cone shortens.

The Tiny-tier roadmap in
[`docs/core-family.md`](../core-family.md) §4.3 already schedules
all three for phase 2B / 2C.
